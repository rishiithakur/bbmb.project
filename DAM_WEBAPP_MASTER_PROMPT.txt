─────────────────────────────────────────────────────────────────────────────
PROJECT OVERVIEW (what the client needs)
─────────────────────────────────────────────────────────────────────────────

This application collects, manages, monitors, and reviews manual dam water
level observations from approximately 50 dam/reservoir monitoring sites
across multiple districts and states. Field staff (Dam Site Users) log in
on mobile phones from field locations and manually enter daily water level
data. Administrators monitor all submitted data through a central dashboard
with GIS map visualization, reports, and alerts.

MODULE 1 — AUTHENTICATION SYSTEM
─────────────────────────────────────────────────────────────────────────────

Build a secure login system with the following specifications:

  1.1  Login page must have username + password fields.
       Show/hide password toggle button.
       "Remember Me" checkbox for session persistence.

  1.2  Two user roles must exist:
         ADMIN USER   — full system access, monitoring, reports, user mgmt
         DAM SITE USER — restricted access, only their assigned site

  1.3  After login, redirect:
         Admin     → Admin Dashboard
         Site User → Their site's Data Entry form

  1.4  Session timeout after 30 minutes of inactivity.
       Show warning popup 5 minutes before timeout.

  1.5  Failed login attempts: lock account after 5 consecutive failures.
       Admin can unlock accounts.

  1.6  All passwords stored using Django's built-in password hashing.
       NEVER store plain text passwords.

  1.7  Implement CSRF protection on all forms.
       Implement SQL injection prevention via Django ORM (no raw queries).

  1.8  Maintain login audit log:
         - Username, login datetime, logout datetime, IP address, status

MODULE 2 — SITE MASTER (Dam/Reservoir Location Registry)
─────────────────────────────────────────────────────────────────────────────

Admin-only module. Store and manage all 50 monitoring site records.

  Database table fields (all required unless marked optional):
    - site_id            (auto-generated unique ID)
    - site_name          (text, e.g. "Bhakra Dam Site 1")
    - site_code          (unique short code, e.g. "BHK-01")
    - dam_reservoir_name (text)
    - district_name      (text)
    - state_name         (dropdown — Indian states list)
    - river_name         (text)
    - latitude           (decimal, e.g. 25.4568)
    - longitude          (decimal, e.g. 77.3214)
    - full_reservoir_level    (numeric, in meters)
    - minimum_draw_down_level (numeric, in meters)
    - danger_level            (numeric, in meters)
    - warning_level           (numeric, in meters)
    - gauge_type         (text, optional)
    - gauge_details      (text, optional)
    - site_status        (Active / Inactive)
    - created_by         (admin username)
    - created_date       (auto timestamp)
    - updated_date       (auto timestamp)

  Admin capabilities:
    - Create new site record
    - Edit/update existing site
    - Activate or Deactivate site (soft delete — never hard delete)
    - Search sites by name, code, district, state, river
    - View all sites in a paginated table (20 per page)
    - Export site list to Excel/CSV

  All active site locations must appear as markers on the Leaflet GIS map.

─────────────────────────────────────────────────────────────────────────────
MODULE 4 — DATA ENTRY FORM (Field Observation Entry)
─────────────────────────────────────────────────────────────────────────────

This is the most critical module. Dam Site Users access this on mobile phones.
UI must be clean, large-touch-friendly, and work on 360px–414px screen widths.

  4.1  FORM FIELDS (all data to be captured):

    SECTION A — Basic Info (auto-filled, read-only for user)
      - Site Name         (from user's assigned site)
      - Site Code         (from user's assigned site)
      - Dam / Reservoir   (from user's assigned site)
      - Observer Name     (from logged-in user's full name)

    SECTION B — Observation Details (user fills these)
      - Observation Date  (date picker, default = today, NO future dates allowed)
      - Observation Time  (time picker, default = current time)

    SECTION C — Water Level Data
      - Current Water Level (numeric, meters, 2 decimal places)
        → Auto-compare with danger_level and warning_level from Site Master
        → Show color alert inline: GREEN / YELLOW / RED
      - Full Reservoir Level (auto-filled from Site Master, read-only)
      - Minimum Draw Down Level (auto-filled, read-only)
      - Danger Level       (auto-filled, read-only, shown for reference)

    SECTION D — Inflow / Outflow / Storage
      - Total Inflow       (numeric, cumecs or MCM — specify unit label)
      - Total Outflow      (numeric)
      - Live Storage       (numeric, MCM)
      - Dead Storage       (numeric, MCM, optional)
      - Percentage Full    (auto-calculated: Live Storage / Full Capacity × 100)

    SECTION E — Rainfall
      - Rainfall Today     (numeric, mm)
      - Cumulative Rainfall (numeric, mm, optional)

    SECTION F — Gate Operations
      - Gates Open?        (Yes / No radio button)
      - If Yes → Number of Gates Open (numeric)
      - Gate Opening Height (numeric, meters, appears only if Gates Open = Yes)
      - Discharge Through Gates (numeric, cumecs)
      - Spillway Discharge (numeric, optional)

    SECTION G — Field Conditions
      - Weather Condition  (dropdown: Clear / Cloudy / Rainy / Foggy / Stormy)
      - Water Quality      (dropdown: Good / Turbid / Discolored / Flood Water)
      - Site Condition Remarks (textarea, max 500 chars)
      - General Remarks    (textarea, max 500 chars)

    SECTION H — Photos & Location
      - Upload Site Photos (allow up to 5 images from device camera or gallery)
        → Compress images client-side before upload (max 1MB each)
        → Show thumbnail preview before submission
      - GPS Latitude  (auto-captured from browser geolocation API, editable)
      - GPS Longitude (auto-captured from browser geolocation API, editable)
      - If GPS unavailable → show message "Location not captured" and allow manual entry

  4.2  VALIDATION RULES (enforce strictly):
    - Observation Date: cannot be future date
    - Current Water Level: required, must be numeric, must be > 0
    - Inflow, Outflow, Storage: numeric only, cannot be negative
    - Rainfall: numeric only, cannot be negative
    - If Gates Open = Yes → Gate height and discharge are mandatory
    - Duplicate check: warn if record for same site + same date already exists in Draft or Final
    - Show clear red error messages below each invalid field
    - Do NOT allow form submission if mandatory fields are empty

  4.3  TWO-STAGE SAVE WORKFLOW (critical business logic):

    STAGE 1 — DRAFT SAVE:
      - "Save as Draft" button saves record with status = DRAFT
      - User can come back, edit, update, or DELETE the draft
      - Draft records are ONLY visible to the field user who created them
      - Draft records do NOT appear in Admin Dashboard
      - Draft auto-save every 5 minutes (save silently in background)

    STAGE 2 — FINAL SUBMIT:
      - "Final Submit" button changes status from DRAFT → FINAL
      - Capture final_submission_datetime (server timestamp, not client)
      - After Final Submit: record is LOCKED — user CANNOT edit or delete
      - Show confirmation popup: "Are you sure? Once submitted, you cannot edit."
      - After successful final submission → show success message with
        submission reference number

    STATUS VALUES: DRAFT | FINAL

  4.4  DRAFT MANAGEMENT:
    - User's home screen shows their draft records in a list
    - Each draft shows: Date, Site, Water Level, Last Edited Time
    - Tap draft → open form pre-filled with saved data → edit → re-save or finalize
    - Delete draft button (with confirmation dialog)

MODULE 7 — REPORTS MODULE
─────────────────────────────────────────────────────────────────────────────

Admin-only. All reports downloadable. Do NOT show data in browser popups.
Generate file server-side and trigger download.

  REPORT 1 — Daily Observation Report
    Parameters: Select Date
    Content: All finalized observations for selected date across all sites
    Columns: Site Code | Site Name | District | Water Level | Inflow |
             Outflow | Storage | Rainfall | Gates | Observer | Time

  REPORT 2 — Site-Wise Report
    Parameters: Select Site + Date Range
    Content: All finalized records for that site within date range
    Include: Site info header, observation table, trend summary

  REPORT 3 — Monthly Summary Report
    Parameters: Select Month + Year
    Content: Day-wise summary for each site — water level trend
    Include: Max/Min/Average water level per site

  REPORT 4 — Missing Observations Report
    Parameters: Date Range
    Content: Sites that did NOT submit observation on each date in range
    Format: Site Name | Missing Dates | Last Observation Date

  REPORT 5 — User Activity Report
    Parameters: Date Range
    Content: Login history, number of records submitted per user
    Columns: Username | Site | Login Count | Records Submitted | Last Active

  REPORT 6 — Reservoir Condition Report
    Parameters: Date Range
    Content: Sites at Warning/Danger level during selected period
    Include: Dates, water levels, status, comparison with danger level

  All reports: Export as Excel (.xlsx) AND PDF AND CSV
  Use openpyxl for Excel, ReportLab for PDF, Python csv for CSV.
  Add report header with: Logo placeholder, Report Title, Generated Date,
  Generated By (admin username), Date Range.

MODULE 9 — SECURITY REQUIREMENTS
─────────────────────────────────────────────────────────────────────────────

  - HTTPS/SSL enforced on all pages (via GCP managed SSL)
  - All Django views require @login_required decorator
  - Role-based access: Admin views protected with @permission_required
  - CSRF tokens on every POST form
  - No raw SQL queries — use Django ORM exclusively
  - Input sanitization on all user-supplied text fields
  - File upload validation: allow only .jpg, .jpeg, .png formats
    Max file size 5MB per upload. Virus scan not required (client decision).
  - Rate limiting on login endpoint (max 5 attempts per IP per 10 minutes)
  - Django SECRET_KEY stored in environment variable, not in code
  - Database credentials stored in GCP Secret Manager or .env file
  - DEBUG = False in production settings
